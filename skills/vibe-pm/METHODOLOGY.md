# vibe-pm 方法论

> 这是 vibe-pm 的"为什么这么做"。
> 想知道"具体怎么做"看每个 skill 的 SKILL.md。

---

## 一、核心命题

**Vibe Coding 不是传统开发的简化版，是一种全新的工作模式。**

| 维度 | 传统开发 | Vibe Coding |
|---|---|---|
| 谁动手 | 工程师写代码 | PM 用自然语言指挥 AI 写代码 |
| PRD 给谁看 | 工程师 + 设计师 | **AI Agent + 人** |
| 写法 | 详尽长文档 | 结构化指令包 |
| 验收 | QA 测试 + 用户反馈 | 评测集 + 实时观察 |
| 节奏 | 周/月 | 小时/天 |
| 失败成本 | 高（团队资源） | 低（重写一次成本极低） |

**因此**：传统的 PM 工作流（BRD → MRD → PRD → 设计 → 工程 → 测试）放到 vibe coding 场景下会过度仪式化、错失速度优势。

vibe-pm 的目标：**保留 PM 的判断力（What 和 Why），删掉中间环节，让 AI 接管 How。**

## 二、7 条不可妥协的原则

### 1. 完成定义优先于功能定义
**反例**：PRD 写了 30 项功能但没说"什么算 done"——开发到一半，"完整版本"被反复重新定义。
**正例**：先用一句话说"用户能在浏览器里和 3 个角色完成一次反思讨论"——后续所有"要不要做 X"都有锚点。

### 2. MVP 砍刀比 PRD 更值钱
**反例**：详细 PRD 写了 20 页，工程量评估出来要 3 个月。
**正例**：先列"V1 不做的 5 件事"，每砍一刀，工程量直接减 30%。**砍刀是稀缺技能**。

### 3. 真实世界的样子 > 文档里的描述
**反例**：用户读 PRD 觉得对，看到 demo 后说"不是这样的"。
**正例**：UI 早期用 mockup 给用户看；prompt 写完直接拿真 API 跑一次。**画面比文字稳，实测比假设稳**。

### 4. 闭环反馈是 AI 产品的命脉
**反例**：上线后没有 bad case 收集机制——AI 答崩了你不知道。
**正例**：每个 AI 输出都有评测集 + 人工抽检 + 自动反馈通道。**没有闭环 = 开盲盒**。

### 5. 文档不是越多越好，而是要"可被引用"
**反例**：写了详细的 PRD 文档，但 AI 写代码时没人提醒它去读哪一段。
**正例**：PRD 拆成 5 个独立文件，AI 写 schema 时只读 03-contracts.md。**模块化 > 长文**。

### 6. 决策的颗粒度和时间感要匹配
**反例**：用做大型项目的方法论讨论一个一晚上就能跑通的小工具。
**正例**：先选"项目档位"（个人原型 / 小团队 / 中型 / 大型），再决定写哪些文档。**别用大炮打蚊子**。

### 7. memory（人和项目的偏好）要随手沉淀
**反例**：和 AI 聊了 50 轮，攒了一堆"用户喜欢翡翠绿/讨厌长篇 PRD"，下次新会话又冷启动。
**正例**：里程碑式存档——任何"这个用户偏好""这个项目特点"都立刻写到 memory。**第一次省力是关键**。

## 三、反例库（同事最常踩的坑，按真实案例积累）

> **每用一次 skill，把翻车的地方加进来。反例 > 正例。**

### Case 001: "完整版本"被反复重新定义
**场景**：圆桌讨论 AI 工具项目
**踩坑**：用户最初说"完整版本一天内"，被理解成"完整功能"。在做了大量设计后，用户实际想要的是"一个 HTML 文件能跑起来"。
**根因**：没在第一轮锁定"完成的可观察标志"
**避坑 skill**：`vibe-pm-define-done`
**关键追问**：Q2"完成那一刻的画面是什么？请描述一个 30 秒能演示的场景"

### Case 002: 写完代码不自测就交付
**场景**：单文件 HTML 圆桌工具
**踩坑**：1500 行 HTML 一次写完直接给用户，结果 `renderMessages` 函数有 DOM 操作 bug，用户发送消息没反应。
**根因**：交付前没用浏览器手动跑一遍 happy path
**避坑 skill**：（规划中）`vibe-pm-engineering-review` —— 闭环反馈维度
**关键检查**：交付前必须自测 happy path，不能用"应该能跑"替代

### Case 003: prompt 设计漏洞导致 agent 集体跳过
**场景**：圆桌的 3 个 AI 角色（芒格/阿德勒/卡帕西）
**踩坑**：在 system prompt 里写"如果你没有补充就跳过"，结果模型过于谦虚，连用户 @ 强制点名都还在跳过
**根因**：prompt 写完没用真 API 验证边缘场景
**避坑 skill**：（规划中）`vibe-pm-ai-eval`
**关键修法**：(1) 强制响应模式 + 禁用回避词清单 (2) 软跳过自动重试

### Case 004: 浏览器缓存导致改了代码以为没生效
**场景**：圆桌单文件 HTML
**踩坑**：用户 Cmd+R 软刷新仍然看到旧版本，以为代码改了无效
**根因**：HTML 模板没默认带 no-cache meta
**避坑模板**：在 SPA HTML 模板里默认加 cache-control meta 标签

### Case 005: 设计文档断在半路
**场景**：圆桌项目设计阶段
**踩坑**：brainstorming 走到第 6 段时用户说"立马"，spec 文档就没写了。后面修 bug 时没有任何"规格说明"可以回看。
**根因**：流程被打断时跳过了"留下书面 anchor"这一步
**避坑 skill**：`vibe-pm-prd` 的 4 阶段交互协议（即使被打断也要留下 M1 + M5）

> （持续积累中。每个翻车都是一个反例。）

## 四、跨平台适配说明

vibe-pm 的方法论与平台无关。具体到 web / iOS / Android / 小程序 / 桌面，差异在于：

| 模块 | 平台无关 | 平台相关 |
|---|---|---|
| 完成定义卡（Q1-Q5） | ✅ 完全通用 | — |
| PRD M1 身份卡 | ✅ 通用 | "平台"字段填具体值 |
| PRD M2 场景脚本 | ✅ 通用 | iOS/Android 时第 5 段改成 "storyboard 截图脚本" |
| PRD M3 三色清单 | ✅ 通用 | — |
| PRD M4 数据契约 | ✅ 通用 | iOS 加"导航栈/页面状态"；小程序加"小程序生命周期" |
| PRD M5 验收脚本 | ✅ 通用 | iOS/Android 加 "device/orientation" 维度 |

**结论**：方法论本体跨平台通用，只在 M2/M4/M5 的子段落里加平台特定字段即可。

## 五、给团队部署的建议

### 5.1 怎么发给同事
**不要**直接发 markdown 文件，而是：
1. 让每个 PM 把这个 vibe-pm/ 文件夹拉到自己的 `~/.claude/skills/`
2. 让 PM 第一次使用时，对 Claude 说"vibe-pm 是什么"
3. 让 PM 用 skill 走完一个真实小项目（建议从个人小工具开始）
4. 复盘 + 把踩坑加到 `METHODOLOGY.md` 的反例库

### 5.2 怎么持续迭代
- **触发记录**：每次用 skill，留一条"触发场景 + 是否有效"
- **每月复盘**：哪些 skill 没人用？哪些被乱用？
- **季度合并 / 修剪**：避免 skill 库无限膨胀
- **反例最有价值**：踩坑当场加 case，比事后回忆准 100 倍

### 5.3 健康指标
skill 总数应该 **先增后减** ——
- 增长期：发现期，每次踩坑 → 新 case / 新 skill
- 平稳期：稳定运行
- 减少期：成熟期，发现某些 skill 重复 → 合并

## 六、什么不在 vibe-pm 范围内

明确**不**做的事，和 PRD 的"红色清单"同源思路：

- ❌ **传统 PRD 工作流**——给开发同事看的 PRD 请用 `prd-generator` 等其他 skill
- ❌ **代码教学**——vibe-pm 不教 PM 写代码，只教 PM 让 AI 写代码
- ❌ **大型企业项目流程**——vibe-pm 不替代 BRD/MRD/调研报告/合规审查；这些场景请用传统流程
- ❌ **需要复杂团队协作的项目**——vibe-pm 默认"PM + AI"两人组合，多角色协调请走传统 PMO
