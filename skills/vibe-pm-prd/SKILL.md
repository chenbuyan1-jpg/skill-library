# vibe-pm-prd — Vibe Coding PRD 撰写

> 这不是给人看的 PRD，是给 AI 执行的"指令包"。
> **衡量它好坏的唯一标准：把这份 PRD 丢给 AI，AI 能不能直接产出一个能跑、能验收、不跑偏的 V1。**

---

## 触发条件

**触发**：
- 用户明确处于 vibe coding 工作流
- 关键词："vibe coding PRD / 让 AI 写一个 / 写 PRD 给 AI 用 / 把想法变成 AI 能干的活"
- `/vibe-pm-prd` 显式调用

**不触发**：
- 用户说"我们研发团队要做"——这是传统 PRD（用 `prd-generator`）
- 用户说"给开发同学看的需求"——传统 PRD
- 用户还在头脑风暴、未到撰写阶段——用 `vibe-pm-define-done`
- 用户已有完整 PRD 想 review——用 `vibe-pm-doc-audit`（规划中）

> **触发的核心判据**：输出物是给 AI 直接落地代码，不是给开发同事讨论。

---

## 核心原则

1. **结构化 > 流畅**——AI 解析标记结构比读流畅的人话快
2. **具体 > 抽象**——例子比形容词有用 100 倍，反例比正例更有用
3. **可验证 > 完整**——没有验收脚本的需求，等于没需求

---

## 4 阶段交互协议（严格执行）

```
阶段 0: 触发判定
   ↓ 是 vibe coding 场景
阶段 1: 输入确认
   - 列已知 + 缺什么
   - 用户补全
   - 提问到无新增信息为止
   ↓ user: "齐了"
阶段 2: 架构提议
   - 产品类型判定
   - 5 模块"内容轮廓"
   - 文件目录结构
   - 关键决策清单（待用户拍板）
   ↓ user: "可以"
阶段 3: 逐模块撰写
   M1 → 简短确认 → M2 → 简短确认 → M3 → 简短确认 → M4 → 简短确认 → M5 → 简短确认
   ↓ 全部完成
阶段 4: 终审
   - 跑 22 项 checklist
   - 列未尽事项
   - 用户拍板 / 要求改
   ↓ user: "通过"
完成
```

**铁律**：任何阶段未拍板前，禁止进入下一阶段。

---

## 阶段 1：输入确认

### 检查清单（缺一不可）
- [ ] 完成定义已锁定？（如未做 → 强制转 `vibe-pm-define-done`）
- [ ] 目标用户画像清晰？
- [ ] 平台已定？
- [ ] 是否 AI 产品已定？
- [ ] 范围意向（V1 想覆盖什么）已粗略对齐？
- [ ] 时间感（紧急程度）已对齐？

### 输出格式
```
## 我已知的信息
- 项目: [简述]
- 用户: [简述]
- 平台: [简述]
- AI 部分: [是 / 否 / 待定]
- 时间感: [简述]

## 我还需要确认
- [ ] 问题 1
- [ ] 问题 2
...

请逐个回答，或告诉我"齐了"我会按已知信息推进。
```

---

## 阶段 2：架构提议

### 输出格式
```
## 产品类型判定
- 类型: [AI 内容生成 / AI 决策辅助 / 工具型 / 信息聚合 / ...]
- 核心交互模式: [对话 / 表单 / 看板 / 流程 / ...]

## 5 模块内容轮廓
- M1 身份卡: 大致会写 [一句概括]
- M2 场景脚本: 计划写 [N] 个场景，重点是 [...]
- M3 三色清单: 绿色 [N] 项，红色重点排除 [...]
- M4 数据与契约: 主要实体 [...]，AI 产品则有 [prompt 草稿要点]
- M5 验收脚本: 核心 happy path [N] 条，失败兜底 [N] 类

## 文件结构
[标准目录树]

## 我需要你拍板的关键决策
- [ ] 决策 1: 选 A 还是 B（我推荐 A，理由...）
- [ ] 决策 2: ...

确认后我开始撰写。如果你想调整哪一项，直接说。
```

## 阶段 3：逐模块撰写

每个模块用 `references/` 下的对应模板：
- M1 → `references/00-identity.md`
- M2 → `references/01-scenarios.md`
- M3 → `references/02-scope.md`
- M4 → `references/03-contracts.md`
- M5 → `references/04-acceptance.md`

### 撰写铁律
- 每完成一个模块，**只输出该模块文件 + 一句话简短确认**
  > "M1 已生成。要进 M2 吗？或者你想先调整 M1？"
- **禁止一次性把 5 个模块全 dump 出来**

## 阶段 4：终审

### 输出格式
```
## 撰写完成。请确认以下：

### Checklist 通过情况（22 项，详见 references/checklist.md）
✅ M1 (3/3)
✅ M2 (4/4)
⚠️  M3 (2/3) — 缺少: 红色清单第 4 项的"原因"
✅ M4 (4/4)
✅ M5 (4/4)
✅ 跨模块 (4/4)
通过率: 21/22

### 关键决策回顾
- 我们当时选了 [...]，理由 [...]
- 待你后续验证的假设: [...]

### 未尽事项
- 我无法确定的: [...]
- 留待 V2 的: [...]

### 完成标志（M1 中锁定的）
[一句话]

请回复:
- "通过" → PRD 锁定
- "改 X" → 我改 X 模块
- "整体重来" → 回到阶段 2 重新提议
```

## 5 个必写模块（缺一不可）

| # | 模块名 | 一句话定义 | 必写理由 |
|---|---|---|---|
| **M1** | 项目身份卡 | 一张卡片说清"我是什么、给谁用、解决什么、不解决什么" | AI 没有"北极星"就会朝多个方向开火 |
| **M2** | 场景脚本集 | 用 30 秒可演示的"给定→操作→结果"脚本描述 3-5 个核心场景 | AI 需要具体的行为目标，抽象描述会被它过度发挥 |
| **M3** | 范围三色清单 | 绿（V1必做）/ 黄（V1可选）/ 红（V1禁止）三类功能 | 不写"红色"就会无限滑坡 |
| **M4** | 数据与契约 | 数据模型 + 接口契约 +（AI 产品）prompt 契约 | AI 需要结构化的内部规格 |
| **M5** | 验收脚本 | 机械可判的 given/when/then，含成功+失败两类 | 没有验收 = AI 自我宣布"完成"，不可控 |

> **顺序就是优先级**。M1 没写好，后面写得再细都白搭。

## 输出目录结构（解耦）

```
PRD/
├─ index.md              ← 一页索引
├─ 00-identity.md        ← M1
├─ 01-scenarios.md       ← M2
├─ 02-scope.md           ← M3
├─ 03-contracts.md       ← M4
├─ 04-acceptance.md      ← M5
├─ evals/                ← AI 产品独有
│  ├─ eval-set.jsonl
│  └─ bad-cases.jsonl
└─ CHANGELOG.md
```

**精髓**：让 AI 写代码时**只读相关文件**：
- 写 schema → 只读 `03-contracts.md`
- 写 prompt → 只读 `03-contracts.md` 4.4 + `evals/`
- 写测试 → 只读 `04-acceptance.md`

## Skill 异常处理

| 情况 | 处理 |
|---|---|
| 用户要求跳过阶段 1 直接写 | 拒绝。"我无法在缺信息时写出能跑的 PRD。请先回答 X / Y / Z" |
| 用户在阶段 3 中途要大改前面模块 | 接受。回到阶段 2 刷新架构后从对应模块继续 |
| 用户给的信息明显不是 vibe coding | 退出 skill。"听起来是给开发同学的 PRD，建议用 prd-generator" |
| 用户没有定义"完成"标志 | 强制转 `vibe-pm-define-done` 后再回来 |

---

## 给同事的 1 分钟启动

1. **第一次用，先读 `references/00-identity.md` ~ `04-acceptance.md`** —— 知道每个模块长什么样
2. **不要追求一次写完**：草稿 → 跑 AI → 改 PRD —— 这是 vibe coding 的真实节奏
3. **写完丢给 AI 之前自己读一遍**：哪句话能两种理解？必修
