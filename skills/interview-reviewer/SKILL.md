# 面试录音复盘 Skill

> 本 skill 已按陈纪富的面试复盘场景做过个人化配置。执行前先读取 `references/user-profile.md`、`references/project-registry.md`、`references/knowledge-vault-index.md`。若用户当次提供的信息与配置冲突，以当次信息为准，并在真实性审核中标注。

**如果你安装了 lark-shared skill** — 开始前先用 Read 工具读取 `../lark-shared/SKILL.md`，其中包含飞书认证和权限处理。没安装则跳过，文档会保存为本地文件。

从面试录音转录文本出发，产出结构化复盘文档，帮助用户真正提升面试能力（不是背答案）。

## 核心原则

1. **教学优于话术** — Part 2「正确答案」不是给一段更好的话术让用户背，而是真正教会知识点的本质。用户背不下AI写的话术，但理解了知识后能用自己的话回答。
2. **个人经历优先** — 复盘不是泛泛评价表达好坏，而是把面试官问题映射到用户真实履历、项目职责、STAR故事、关键数据和目标岗位要求。建议回答必须能解释"为什么是你能这么说"，不能给换个人也能用的模板。
3. **项目文档是 ground truth** — 生成「项目深挖清单」时，答案来自用户自己的项目文档（PRD、架构文档、评测SOP等），不是AI编造。文档没覆盖的如实标注"文档中未覆盖，需要你自己补充"。
4. **聚焦3-5题** — Part 2-4只对答得最差的3-5个关键问题做深度展开。全做会信息过载，用户抓不住重点。
5. **动态读取不缓存** — 简历、项目文档、STAR故事集和 Obsidian 知识库都在持续更新。每次执行必须重新读取最新版本，不使用任何缓存。知识库路径以 [`references/knowledge-vault-index.md`](references/knowledge-vault-index.md) 为准；当前 vault 不是默认的 `原子笔记/沉淀` 结构，不能硬套旧目录。如果没有匹配笔记，用 WebSearch 补充知识。
6. **发布前必须审核** — 所有生成内容在发布前，必须经过三重审核：可用性（用户能不能在真实面试中说出来）、真实性（是否符合用户实际经历）、知识库一致性（是否与用户已有知识体系吻合）。审核不通过的内容必须修正或标注警告后才能发布。
7. **以目标年薪/目标岗位为对标** — 目标岗位、目标薪资和用户风险偏好以 [`references/user-profile.md`](references/user-profile.md) 和当次用户输入为准。所有回答示例、训练任务、自查清单都以目标档位标准设计，并明确标注用户当前 level 评级（初级/中级/高级）和差距。**所有训练任务和自查项必须配套参考答案，不能只列任务不给答案**——清单没有答案 = 没有训练价值。
8. **按薪资档位训练表达** — 面试复盘不仅判断"答得好不好"，还要判断"这个回答值多少钱"。对关键弱题和高频题，必须给出 20万 / 30万 / 35万 / 40万 / 50万+ 年薪档位的回答差异，指出用户当前回答落在哪一档、目标薪资需要补什么能力信号。薪资档位是表达质量和岗位责任的训练标尺，不是承诺市场报价。
9. **先读取用户画像** — 每次执行先读取 [`references/user-profile.md`](references/user-profile.md)，作为用户个人叙事、薪资背景、风险偏好和项目默认数据。若当次用户提供的信息与画像冲突，以当次信息为准，并在真实性审核中标注差异。
10. **固定骨架必须保留** — 输出结构必须遵循 [`references/review-output-framework.md`](references/review-output-framework.md)：保留面试概览、岗位画像、Part 1-5、本场可复用话术、待确认事项、审核状态。不能因为内容少而删掉目录；无信息时写"本场未出现/暂无"，但保留位置。

## 工作流

```
Phase 0: 输入收集
  读取 references/user-profile.md（用户画像与默认目标）
  读取 references/review-output-framework.md（固定输出骨架与格式约束）
  转录文本 + 目标JD + 讨论了哪些项目 + 公司名 + 日期 + 面试官角色
  + 当前薪资/期望薪资/目标年包（可选）
    ↓
Phase 1: 转录处理 → QA配对（Part 1）
  清洗口语 → 提取岗位画像/口述JD → 切分QA对 → 提取临别反馈/金句 → 选出3-5个关键弱题
    ↓
Phase 2: 知识解析 → 正确答案教学 + senior对标（Part 2）
  [动态] 读Obsidian知识库 + 项目文档 + 简历
  每题输出: 正确理解 / 在你项目中 / 关键数据 / 行业实践
           + 目标档位回答示例 + 你当前level评级
           + 薪资档位表达梯度（20/30/35/40/50万+）
    ↓
Phase 3: 思维路径生成（Part 3）
  每个弱题建模 Step1→Step2→Step3→Step4 条件反射路径
    ↓
Phase 4: 追问防御链（Part 4）
  每个弱题延伸2-3个追问 + 意图 + 关键词
    ↓
Phase 5: 差距分析 + 训练闭环（Part 5）
  5a 项目深挖清单 | 5b 知识盲区清单 | 5c JD匹配映射
  5d 本周训练任务（每条配参考答案+验收标准）
  5e 下次面试前自查项（每条配标准答案+答案来源）
  5f 薪资档位/岗位风险判断（可选但推荐）
  5g 本场可复用话术（新增题/补充原话/提问套路）
    ↓
Phase 6: 三重审核（发布前质量关卡）
  可用性审核 → 真实性审核 → 知识库一致性审核
  不通过的内容修正或标注警告
    ↓
Phase 7: 组装文档 → 发布（飞书 or 本地文件）→ 返回链接
```

## Phase 0：输入收集

需要收集6个输入。如果用户没有一次性提供全部，**用纯文本方式在对话中提问**，让用户自己输入回答。

**严禁使用 AskUserQuestion 工具弹窗选择**——用户希望保持对话流的自然性，自己用文本回答即可。直接在对话里列出问题清单，等用户回复。

### 输入清单（按询问顺序）

**询问顺序原则：先问短的元信息，最后问长的转录文本**——避免用户先粘贴一大段转录文本后还要继续被问其他问题。

| 顺序 | 输入 | 必填 | 接受格式 | 说明 |
|------|------|------|---------|------|
| 1 | 公司名 | 是 | 文本 | 用于文档标题 |
| 2 | 讨论的项目 | 是 | 项目名 / 飞书链接 / 本地文件路径 / 自由描述（任意组合，可多个） | 开放式询问，不预设选项 |
| 3 | 面试官角色 | 是 | HR / HM直属leader / 同级技术伙伴 / 高管(VP/CTO) / 不确定 | 决定Phase 1-4的诊断重点和追问预测 |
| 4 | 目标JD | 推荐 | 粘贴文本 / 图片路径 / URL | 没有则跳过Part 5c岗位匹配 |
| 5 | 面试日期 | 否 | YYYY-MM-DD | 默认当天 |
| 6 | 转录文本 | 是 | 粘贴文本 / 本地文件路径(.md/.txt) / 飞书文档链接 | **最后问**，来自录音转录或手动整理 |
| 7 | 当前薪资/期望薪资/目标年包 | 否 | 如 25K*13 / 期望30-35万 / 不方便说 | 用于判断回答档位和谈薪策略；用户没提供时根据JD薪资范围推断并标注"推断" |

### 面试官角色考察重点（用于Phase 1-4调整诊断重点）

| 面试官角色 | 考察重点 | 诊断侧重 | 追问类型 |
|-----------|---------|---------|---------|
| HR | 动机、稳定性、薪资预期、文化匹配 | 离职原因表达、职业规划逻辑 | 软性追问（家庭/异地/期待） |
| HM 直属leader | 执行力、专业能力、项目细节 | 项目深度、技术理解、协作经验 | 项目细节追问（trade-off/数据/决策） |
| 同级技术伙伴 | 技术深度、概念准确性、行业认知 | 知识准确度、术语规范、广度 | 概念追问（原理/对比/最新进展） |
| 高管 (VP/CTO) | 战略视野、判断力、行业格局 | 商业理解、宏观判断、组织能力 | 战略追问（为什么这个方向/如何规模化） |

### 项目询问（开放式，纯文本提问）

**不预设选项、不弹窗、不用AskUserQuestion**——直接在对话中用文本提问：

```
"这场面试中你聊到了哪些项目？请告诉我：
- 项目名称（如果有）
- 项目链接（飞书文档/本地文件路径/Obsidian笔记路径都行）
- 或者直接描述（项目背景、你的角色、做了什么）

可以提供一个或多个项目。"
```

等用户用文本回复后，按下方规则处理。

### 用户回复的处理逻辑

用户可能以以下任意方式回复，skill需要全部支持：

| 用户回复形式 | 处理方式 |
|------------|---------|
| 项目名称（如"小剧场"） | 在 `references/project-registry.md` 查找匹配项，找到则用注册表中的文档路径 |
| 飞书文档链接 | 用 `lark-cli docs +fetch --doc <url> --as user` 读取内容作为项目文档 |
| 本地文件路径 | 用 Read 工具读取（支持md/txt/pdf）|
| Obsidian笔记路径 | 用 Read 工具读取 |
| 项目目录路径 | 用 Bash 的 `ls` 列出目录中的md文件，逐个读取 |
| 自由文本描述 | 用用户描述作为项目背景信息，标注"无项目文档可参考" |
| 混合形式（项目名+链接+描述） | 全部接收，分别处理 |

### 处理规则

1. **优先使用注册表** — 如果用户提的项目名能在 `references/project-registry.md` 找到，自动加载该项目的全部文档（不需要用户重复提供）
2. **链接/路径直接读取** — 用户提供链接或路径时，直接读取内容，不要求用户再选
3. **未注册项目的兼容** — 如果是注册表外的新项目，用户提供的文档作为唯一参考；同时**建议用户更新 project-registry.md**以便后续复用
4. **多项目支持** — 用户可能讨论多个项目，全部读取后在 Part 5 项目深挖清单中分别处理

### 输入验证

- 转录文本至少500字符（真实面试远多于此）
- 如果是飞书链接，用 `lark-cli docs +fetch --doc <url>` 获取内容
- 如果JD是URL，用 WebFetch 获取；如果是图片，用 Read 工具读取

## Phase 1：转录处理（输出 Part 1）

### Step 1.1：清洗口语

去除语气词（嗯、啊、那个、就是说）、重复表述、false starts。**保留候选人的原始语义**——清洗是为了可读性，不是重写。

### Step 1.2：识别角色

- 如果转录有说话人标签（如"发言人1/发言人2"），按标签区分
- 否则按提问/回答模式判断：提问的是面试官，回答的是候选人

### Step 1.3：切分QA对

每个QA对包含：
- **面试官问题**（清理口语后）
- **候选人实际回答**（保留原意）
- **简短诊断**（2-3句）：这个回答好在哪/差在哪，标注严重度

严重度标注：
- 🔴 **关键失分**：事实性错误、完全答不上、严重减分
- 🟡 **有待改进**：方向对但不够深、缺少数据支撑、表述不清
- 🟢 **表现不错**：回答到位，可能有小的改进空间

### Step 1.4：选出3-5个关键弱题

从所有QA对中选出答得最差的3-5个，用于Part 2-4深度展开。选择优先级：

```
事实性错误/完全答不上 > 面试官反复追问仍答不好 > 核心高频题 > 直接对应JD要求的题
```

### Step 1.5：提取岗位画像与面试官强信号

如果面试官在面试中口述岗位、团队、目标、资源、定级、下一轮安排，把它当作"下一轮真实JD"处理，即使用户没有提供正式JD。

必须单独提取：

- **面试官口述的岗位画像**：岗位核心、业务痛点、多端场景、指标要求、团队资源、稳定性要求。
- **临别总结/金句**：面试官最后主动给的反馈、点名的硬伤、定级/推进/下一轮信号。
- **信号判断**：区分"通过信号"、"定级风险"、"岗位风险"。不要只写情绪判断，要写证据句。

## Phase 2：知识解析 → 正确答案教学（输出 Part 2）

这是最核心的Phase。对每个选出的弱题，生成真正的知识教学内容。

### 知识解析策略（三层）

**Layer 1 — 问题分类**

将弱题映射到知识域。按用户在 `references/knowledge-vault-index.md` 中配置的关键词匹配规则进行分类。如果没有知识库索引，按问题主题自行分类。

**Layer 2 — 用户知识库查找**

知识库路径和分类以 [`references/knowledge-vault-index.md`](references/knowledge-vault-index.md) 为准。当前用户的 Obsidian 主要入口是：

1. `10_知识卡片/` — AI应用、方法论、工具使用、创业等轻量卡片
2. `AI产品经理/` — AIGC 图片/视频/分镜提示词资产
3. `20_输出文档/`、`wiki/` — 如有相关专题，再按关键词读取

每个知识域先读标注 `(start here)` 的文件。没有匹配时，在 vault 内按关键词搜索 1-3 个最相关 `.md` 文件。

**如果没有 Obsidian 知识库：** 跳过 Layer 2，直接用 WebSearch 搜索相关知识补充。所有知识标注为 `[来源：外部补充]`。

**补充规则：**
1. 项目相关问题额外读对应项目文档（从 project-registry.md 取路径）
2. **总读取文件控制在8-10个以内**，按与弱题的相关性排优先级
3. 始终按 [`references/project-registry.md`](references/project-registry.md) 读取共享参考中的简历和用户画像
4. 如果后续配置了 STAR 故事集，也作为参考

**Layer 3 — 交叉验证**

- 数据/指标：与简历和项目文档交叉核对，确保一致
- 知识框架：确认在用户vault中存在，不编造用户没学过的框架
- vault中没有的知识：用 WebSearch 补充，但明确标注"外部知识，尚未收入你的vault"

### 每题输出结构

对每个弱题，输出5个子节：

```
### 为什么这是关键失分点
这题为什么影响通过/定级/薪资/岗位匹配。

### 正确理解
这个问题的本质是什么？涉及什么概念/框架/原理？
（来源：Obsidian知识库 or WebSearch）

### 在你的项目中
这个知识点在你的项目中具体是怎么回事？
（来源：项目文档，用项目文档中的真实细节）

### 关键数据解释
如果涉及指标/数据，怎么定义、怎么观测、怎么解释？
（来源：项目文档 + 简历，交叉核对）

### 行业最佳实践
业内领先团队怎么做的？
（来源：知识库 or WebSearch）

### 目标档位回答示例 + 你的Level评级
这是核心模块——以senior标准对标用户当前回答。

### 距 senior 的 gap + 补法
指出具体缺口，并给出一条可练成条件反射的补法。
```

### Senior级回答示例的写法

不是给一段话术让用户背，而是给**结构示范**。包含3个层次：

```
【senior级回答的结构】
开头（破题+定结论）：用一句话给出明确判断，让面试官立刻知道你的立场
中段（数据+论据）：用3-4个具体数据/事实支撑结论，每个数据带来源
收尾（战略视野）：跳出问题本身，谈这个决策对业务/产品/团队的更广泛影响
```

然后基于这个结构，给出**针对本题的senior级范例**：用用户真实项目的数据填充，写一段约150-250字的范例回答。

### Level评级标准

对用户在本题的实际回答，给出明确评级。以下薪资区间请根据你的目标岗位和城市自行调整：

| Level | 表现特征 | 参考薪资区间 |
|-------|---------|------------|
| 🔴 初级 | 答错、答不上、答得空洞 | 目标薪资的50-65% |
| 🟡 中级 | 答对方向但缺数据/缺trade-off/缺战略视野 | 目标薪资的65-85% |
| 🟢 高级（senior） | 有结构、有数据、有战略、能hold住追问 | 目标薪资的85-110%（target区间） |
| 🌟 专家级 | 能站在更高视角批判性思考，提出原创见解 | 目标薪资的110%+ |

评级时必须明确指出：
1. 用户当前在哪一档
2. 离目标岗位/目标薪资档位还差什么具体维度
3. 这个gap可以通过什么训练补上

### 年薪档位回答梯度

对关键弱题、核心项目题、高频HR题（自我介绍/离职原因/职业规划/薪资期望/为什么选我们）必须给出"回答值多少钱"的梯度判断。需要输出该模块时，读取 [`references/salary-level-ladder.md`](references/salary-level-ladder.md)，按其中的 20万 / 30万 / 35万 / 40万 / 50万+ 档位标准生成当前档位、目标档位、档位差异和冲刺版回答。

## Phase 3：思维路径生成（输出 Part 3）

对同样的3-5个弱题，生成条件反射训练路径。

### 格式

```plaintext {wrap}
听到"{面试官的问题}" →

Step 1（先定义/给结论）：{听到这个问题后，第一反应应该想什么}
Step 2（说原因/trade-off）：{为什么这么做/不这么做}
Step 3（对比其他方案）：{还有哪些方案，为什么不选}
Step 4（回到你的场景和结果）：{在你的项目中怎么做的，结果如何}
```

### 要求

- 每步用一句话描述，不写长段
- 目标是训练条件反射，不是提供完整答案
- 思维路径应参考知识库中的框架
- Step 4必须映射到用户的真实项目经验

## Phase 4：追问防御链（输出 Part 4）

对同样的3-5个弱题，列出面试官最可能的追问。

### 每个追问包含

- **追问问题**（面试官可能怎么问）
- **追问意图**（他在测试什么能力/知识）
- **思路关键词**（不是完整答案，是你该想到的2-3个关键点）

### 追问来源

1. 面试中实际发生的追问（从转录中提取）
2. 按问题主题推导的高概率追问
3. 参考已有追问题库（如项目文档中的面试追问题库）

每个弱题列2-3个追问即可，不要过多。

## Phase 5：差距分析（输出 Part 5）

### 5a. 项目深挖清单

对面试中讨论到的每个项目：

1. 读取该项目的完整文档集（从 project-registry.md 取路径列表）
2. 对照面试中答得不好的问题，检查：
   - 项目文档中已覆盖 → 标注"你的文档里有，去复习：{文件名}"
   - 项目文档中未覆盖 → 标注"文档未覆盖，你需要自己想清楚并补充"
3. 输出为问题清单格式，每条是一个用户必须能用自己的话回答的问题

### 5b. 知识盲区清单

对面试中暴露的每个知识缺失：

| 概念 | 一句话解释 | 为什么你需要懂 | 知识库参考 |
|------|----------|-------------|-------------|
| {概念名} | {解释} | {与面试/岗位的关系} | {vault中的文件路径，or "尚未收录"} |

按知识域分组。

### 5c. 岗位匹配映射

**前提：用户提供了目标JD。没有JD则跳过本节。**

读取简历和项目文档，逐条比对JD要求：

| JD要求 | 匹配度 | 你的经验证据 | 面试中的表现 | 下次怎么讲 |
|--------|--------|------------|------------|----------|
| {JD条目} | ✅直接覆盖 / 🔄迁移论证 / ❌空白 | {来自简历/项目文档} | {这场面试中有没有展示} | {改进建议} |

末尾总结核心差距（按紧急度排序的3个待补项）。

---

### 5d. 本周训练任务（每条配参考答案）

基于本场面试暴露的问题，给出3-5个**本周内可执行的训练任务**。每条训练任务必须包含：

1. **任务描述** — 要训练什么具体能力
2. **参考答案** — 直接给出senior级标准回答内容（不是话术，是结构示范+关键论据+数据支撑）。基于用户真实项目和知识库构建，**禁止编造**。如果某个数据/细节用户文档中没有，用 `⚠️ 此处需用户补充真实数据：xxx` 占位
3. **验收标准** — 用户怎么自检训练是否达标

### 5e. 下次面试前自查项（每条配标准答案）

针对**简历上的每个数据/项目细节**，列出下次面试前必须能答出的关键问题。每条自查必须包含：

1. **自查问题** — 面试官最可能直接问的版本
2. **标准答案** — 可以直接说出口的完整回答（约100-200字）。**基于用户真实项目文档和知识库**，标注答案来源
3. **答案来源** — 这个答案来自哪份文档/哪条知识库笔记
4. **数据补充警告** — 如果答案需要的真实数据用户文档里没有，用 `⚠️ 此处需用户补充真实数据：xxx` 明确标注

**核心约束：** 5d 和 5e 中的每个答案都必须能追溯到用户的真实项目或知识库。AI不能为了"看起来完整"而编造细节。文档中没有的，宁可空着标警告，也不能编造。

### 5f. 薪资档位/岗位风险判断（推荐输出）

当用户提供当前薪资、期望薪资、JD薪资范围，或面试中出现薪资/职级/岗位责任相关信息时，必须输出本节。用户未提供薪资时，也可以根据JD薪资范围和面试表现做"推断版"，并明确标注"基于JD推断，非市场报价"。

读取 [`references/salary-level-ladder.md`](references/salary-level-ladder.md) 输出三块内容：

1. **回答档位总评** — 当前表达约等于哪个年薪档，目标岗位大概需要哪个档，核心差距是什么。
2. **高频题档位话术** — 至少选3个最影响薪资的题（自我介绍、核心项目、离职原因、职业规划、为什么选公司、薪资期望等），给出当前档位、目标档位需要补的能力信号和冲刺版回答。
3. **岗位是否值得去** — 当面试暴露组织/岗位风险时，按成长、权限、目标、资源、薪资五个维度判断"值得去/谨慎去/不建议去"，并给出必须追问的问题。

### 5g. 本场可复用话术

把本场沉淀成可复用资产，输出三类：

1. **新增题** — 本场新出现且值得进话术库的问题。
2. **已有题补充原话** — 自我介绍、离职原因、核心项目等高频题的新表达。
3. **提问套路/面试官风格** — 如"边问边教型leader"、"挤牙膏式深挖"、"拿简历逐字对质"。写清应对策略。

如果要写入话术库，必须先确认目标路径；未确认时只在报告中输出"本场可复用话术"，不擅自写入其他文件。

## Phase 6：三重审核（发布前质量关卡）

所有 Part 1-5 的内容生成完成后，**必须在发布前执行以下三轮审核**。这是硬性关卡，不能跳过。

### 审核1：可用性审核

**核心问题：用户能不能在真实面试中自然地说出来？**

逐条检查 Part 2-4 的内容：

| 检查项 | 不通过的表现 | 修正方式 |
|--------|------------|---------|
| 语言自然度 | 读起来像论文/教材，不像人在面试中会说的话 | 改成口语化表达，保留核心知识点但换成对话语气 |
| 知识密度 | 单个回答塞了太多概念，面试中根本说不完 | 拆分为核心回答 + 追问时再展开的备用内容 |
| 思维路径可操作性 | Step过于抽象（如"分析trade-off"），听完不知道具体想什么 | 替换为具体动作（如"想三个词：沉没成本、用户预期、失败分布"） |
| 数据可说出口 | 引用了用户其实记不住的精确数字 | 只保留用户能记住的关键数字（1-2个），其余改为量级描述 |

### 审核2：真实性审核

**核心问题：内容是否符合用户的真实经历？**

逐条交叉核对：

1. **数据核实** — Part 2中提到的所有数字（指标值、百分比、成本数据），必须与简历和项目文档中的数据一致。不一致的标注 `⚠️ 数据待确认：简历写X，这里写Y，请用户核实`
2. **经历核实** — "在你的项目中"部分描述的做法，必须在项目文档中有对应内容。项目文档中没有的，不能编造，必须标注 `⚠️ 文档中未找到此细节，请用户确认是否属实`
3. **角色边界** — 检查是否把别人做的事写成了用户做的。对比项目文档中用户的角色描述，确保归属正确
4. **时间线合理性** — 如果涉及时间（优化周期、上线时间），检查是否和用户的工作时间线吻合

### 审核3：知识库一致性审核

**核心问题：生成的"正确答案"是否和用户已有知识体系吻合？**

对 Part 2 中引用的每个知识框架/概念：

1. **回查知识库** — 重新读取 Phase 2 中引用的知识库文件，确认：
   - 生成内容中使用的术语和知识库中的一致
   - 框架的分类方式和知识库一致
   - 没有引用用户知识库中不存在的框架（如果引用了，标注来源）

2. **标注知识来源** — 每个知识点标注来源类型：
   - `[来源：知识库]` — 来自用户知识库的内容
   - `[来源：项目文档]` — 来自用户项目文档的内容
   - `[来源：外部补充]` — 来自 WebSearch 的外部知识

3. **冲突检测** — 如果生成内容与知识库中的观点/数据存在矛盾，不静默覆盖，而是标注 `⚠️ 与知识库冲突：知识库中说X，此处建议Y，原因是Z，请用户判断`

### 审核输出

- **全部通过** → 直接进入 Phase 7 发布
- **有警告项** → 在文档末尾汇总所有 `⚠️` 标注，并在返回结果中提醒用户检查
- **有严重问题**（如核心数据矛盾、编造经历）→ 暂停发布，先向用户报告问题，等用户确认后再修正发布

## Markdown注意事项

### 本地 Markdown 模式（默认）

- 不使用飞书/HTML 风格标签：禁止 `<callout ...>`、`</callout>`。
- 诊断块用普通 Markdown 表达：`**诊断：**`、`> 诊断内容`、表格、列表。
- 代码块用标准格式：` ```text `，不要使用 `{wrap}` 扩展语法。
- 适配 Obsidian/本地预览：优先用标题、引用块、粗体标签和表格，确保复制到任意 Markdown 编辑器都能正常显示。

### 飞书 Markdown 模式

- **callout内只能放**：文本、列表、代码块
- **callout内不能放**：表格、分割线（---）、标题
- 表格放在callout外面
- 代码块用 ` ```plaintext {wrap} ``` ` 格式展示实际回答和思维路径
- 诊断用callout色彩区分：`light-green`=好 / `light-orange`=待改进 / `light-red`=关键失分

## 特别注意

1. **转录质量参差不齐** — 自动转录对技术术语识别差（Agent/Prompt/RAG等可能被错误转录），处理时做最佳努力修正
2. **上下文窗口管理** — 转录+项目文档+知识库可能很大。Phase 2只加载与弱题相关的文件，不一次性加载全部
3. **新项目不在注册表中** — 如果用户讨论了注册表外的项目，Phase 2-4仍可基于转录内容工作，Phase 5的项目深挖清单标注"无项目文档可参考"
4. **没有JD** — 跳过Part 5c岗位匹配映射，其余部分正常执行
5. **没有 Obsidian 知识库** — 所有知识用 WebSearch 补充，标注为外部来源。核心功能不受影响
6. **不编造经历** — 正确答案教学是教知识，不是替用户编造项目经历。如果用户的项目中确实没做过某件事，教用户怎么诚实地回答"这部分我没有直接负责，但我了解..."
